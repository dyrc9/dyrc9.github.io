RL notes 7 - hierarchical RL and skills

前面写连续控制时，action还是一步控制量；写VLA时，又提到长程任务经常需要hierarchy。这里补一下中间这层：hierarchical RL / skill learning。

最朴素的问题是：如果任务很长，智能体每一步都从最底层动作开始决策，会非常难学。比如“打开抽屉拿杯子放到桌上”，低层动作可能是几百个关节控制命令，但人理解任务时不会按这个粒度想。

人更像是在想：

  • reach drawer
  • pull handle
  • grasp cup
  • move to table
  • release

这些中间动作片段就可以看成skill。

option

Option framework 把一个高层动作定义成三部分：

  • initiation set：哪些状态可以开始这个option
  • intra-option policy：option内部怎么行动
  • termination condition：什么时候结束

粗略写成：

o = (Io, πo, βo)

其中βo(s)表示在状态s结束option的概率。

普通MDP每一步选primitive action；hierarchical RL里，高层policy选option，option内部执行若干步动作，直到触发termination。

直觉上，option把时间尺度拉长了。高层不需要每个timestep都管细节，只需要决定现在用哪个skill。

temporal abstraction

分层的关键是 temporal abstraction。一个skill持续多步，所以高层决策频率更低。

这带来两个好处：

  • 长程任务的credit assignment更短一些
  • 探索可以在skill空间里进行，而不是原始动作空间

比如机器人已经学会grasp和place，那么新任务只需要组合skill，而不是重新从关节动作探索。

但坏处也明显：如果skill本身不好，高层再聪明也没用。skill的边界、启动条件、终止条件都会影响最终表现。

skill discovery

skill可以人工定义，也可以自动发现。

人工定义的skill更可靠，比如导航、抓取、开门。但它依赖工程经验，泛化有限。

自动发现skill常见思路是鼓励多样性或可区分性。比如希望不同skill能把智能体带到不同状态区域，这样高层有更多可选行为。

可以把skill记成一个隐变量z：

at ∼ π(at|st, z)

不同z对应不同skill。训练时鼓励轨迹能反推出z，也就是skill之间要有可区分行为。

不过这类unsupervised skill discovery有个问题：发现出来的skill不一定对任务有用。模型可能学会原地转圈、来回移动这种“可区分但没意义”的行为。

所以实际任务里，skill discovery经常还要和任务奖励、数据先验或人工约束结合。

goal-conditioned policy

另一条常见路线是 goal-conditioned policy：

at ∼ π(at|st, g)

这里g可以是目标位置、目标图像、语言目标或latent goal。

如果policy能根据不同goal行动，那么高层planner只要给出一串子目标。这样skill不一定是离散的option，也可以是连续目标条件下的控制器。

机器人里这很自然。比如高层决定“末端移动到杯子上方”，低层policy负责从当前姿态到目标位姿。

这和VLA里的语言指令也能接起来：语言给高层goal，视觉定位目标，低层执行goal-conditioned control。

hierarchy和planning

分层结构经常和planning放在一起。

如果有world model，高层可以在抽象状态上规划：

  • 当前有哪些对象
  • 每个对象的关系
  • 哪些skill可用
  • 执行skill后状态怎么变

低层不需要知道完整任务，只需要把当前skill执行好。

这有点像 classical planning + learned skill policy 的混合。符号规划负责长程结构，神经策略负责感知和控制。

纯端到端策略当然更简洁，但长程任务里解释性和可调试性会差。分层方法至少让错误更容易定位：是planner选错了，还是skill执行失败。

和具身智能的关系

具身智能里hierarchy几乎绕不开，因为真实任务天然跨时间尺度：

  • 视觉识别是短时刻
  • 抓取是几秒
  • 整理房间是几分钟
  • 和人协作可能更久

一个模型如果每一步都直接输出低层动作，会很难记住任务进度。分层后，高层可以维护任务状态，低层专心做局部控制。

VLA模型里也有类似问题。语言指令可能是高层goal，但动作输出是低层控制。中间如果没有子目标或skill，模型需要自己在隐状态里学出层级结构。

这也是我觉得hierarchical RL仍然有价值的原因：它给了一个很清楚的问题分解。

小结

这条线可以这样记：

  • option：把一个高层动作封装成多步policy
  • temporal abstraction：减少高层决策频率
  • skill discovery：自动找可复用行为，但不一定任务相关
  • goal-conditioned policy：用目标调节低层控制器
  • hierarchy + planning：适合长程具身任务

从RL走到robot policy，action的含义会不断变大：从primitive action，到continuous control，再到skill，最后可能变成语言条件下的子任务。
