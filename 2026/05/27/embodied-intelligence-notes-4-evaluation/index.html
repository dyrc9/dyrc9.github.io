Embodied intelligence notes 4 - evaluation and failure modes

前面几篇主要在写策略怎么学：VLA、diffusion policy、action chunking、hierarchy。这里换个角度：怎么评估一个机器人策略真的好？

在普通ML任务里，评估经常是跑一个test set；机器人不太一样。一次失败可能意味着物体掉了、场景要重置、机械臂进入奇怪姿态，甚至需要人工介入。所以评估本身也是系统的一部分。

success rate 不够

最常见指标当然是 success rate：

但这个指标会丢掉很多信息。比如两个策略都是80%成功：

  • A策略失败时只是没抓起来，可以很快重试
  • B策略失败时会把杯子扫到桌下，需要人工重置

两者的部署价值完全不同。

所以评估时至少还要看：

  • failure severity
  • recovery ability
  • intervention count
  • task completion time
  • trajectory smoothness
  • safety margin

机器人不是只要“最后成功”就行，中间过程也很重要。

intervention

真实机器人实验里，经常会记录 human intervention。比如策略执行到一半卡住，需要人暂停、扶正物体、重置机械臂。

intervention count 有时比success rate更敏感。一个策略可能最后能成功，但每次都要人帮一次，那就不算真正可用。

可以粗略分几类：

  • no intervention：完全自主完成
  • soft intervention：人类给一个提示或重新指定目标
  • hard intervention：需要物理重置环境
  • emergency stop：安全相关中断

不同级别的intervention成本差异很大，不能都算成同一个失败。

distribution shift

具身策略最怕训练环境和测试环境差一点。

常见shift：

  • 光照变化
  • 相机位置轻微移动
  • 物体材质变化
  • 物体初始位置变化
  • 背景杂物变化
  • 机器人关节或夹爪有磨损

有些变化对人来说很小，但对视觉策略可能是很大的分布外输入。

所以评估时最好不要只测一个干净场景。可以人为做一些扰动：

  • object pose randomization
  • distractor objects
  • camera viewpoint shift
  • lighting change
  • partial occlusion

如果策略只在一个固定桌面上成功，意义有限。

failure taxonomy

记录失败时，不要只写 failed。最好拆成类型。

抓取任务里常见失败：

  • detection failure：没找到目标
  • bad affordance：抓取点不可行
  • collision：路径碰到物体
  • slip：抓住后滑落
  • premature release：过早松开
  • timeout：一直犹豫或重复动作
  • recovery failure：失败后不会重试

这些失败对应不同改进方向。detection failure可能要改视觉；slip可能要改夹爪控制或触觉反馈；timeout可能是policy不确定性太高。

分类失败模式的价值是让下一步迭代有方向，而不是盲目加数据。

robustness vs generalization

robustness和generalization有点像，但不完全一样。

robustness更像是在同一任务附近抗扰动：杯子移动一点、光线变暗一点、相机抖一点。

generalization则是跨任务或跨对象：从红色杯子泛化到透明杯子，从方块堆叠泛化到餐具摆放。

一个策略可以很robust但不generalize。比如它在一种杯子上非常稳，但换一个形状就不行。

评估时最好明确自己在测哪一个，不然结果会很含糊。

online adaptation

真实机器人里，策略不一定要一次成功。它可以观察失败，然后调整。

比如抓取失败后：

  1. 重新观察物体位置
  2. 判断是否滑落
  3. 换一个抓取方向
  4. 再试一次

这种 recovery loop 很关键。很多benchmark只看单次attempt，但真实系统更像一个闭环。

如果策略能从失败中恢复，它的有效成功率会高很多。这里就又回到RL和planning：失败不是episode结束，而是新的状态。

safety

安全评估不能靠平均指标。

即使99次成功，1次高速撞桌也不能接受。机器人策略里需要特别关注尾部风险：

  • 最大接触力
  • 最小距离
  • 速度/加速度限制
  • 是否进入不可恢复姿态
  • 是否可能损坏物体

这和RL里的expected return不完全一致。expected return可能会掩盖低概率高损失事件。

因此部署时常常需要额外的 safety layer，例如动作限幅、碰撞检测、workspace限制、急停机制等。

evaluation protocol

一个比较朴素但可复用的评估记录可以包含：

  • task name
  • object set
  • initial state distribution
  • number of trials
  • success definition
  • intervention definition
  • failure categories
  • environment variations
  • safety constraints

这些东西写清楚后，结果才可比较。不然一句“成功率90%”其实信息量很小。

小结

具身智能评估可以先记几个问题：

  • 成功率之外，失败有多严重？
  • 需要多少人类介入？
  • 环境扰动后还能不能做？
  • 失败模式具体是哪类？
  • 策略能不能恢复？
  • 有没有低概率高损失风险？

很多时候，评估不是最后一步，而是下一轮数据收集和模型改进的起点。策略在哪里失败，下一批数据就应该围绕那里补。
