Embodied intelligence notes 6 - robot foundation models and data flywheel

前面写 VLA 时提到过 robot data mixture。这篇把这个问题再往外展开一点：如果想做 robot foundation model，真正难的到底是什么？

直觉上当然是模型要更大、数据要更多。但具身智能里，“更多数据”不是把网页文本继续堆大那么简单。机器人数据带着身体、传感器、控制频率和场景限制，每一条轨迹都很具体。

所以我现在更愿意先把 robot foundation model 看成一个数据系统问题，而不是单纯的模型缩放问题。

what is shared

不同机器人之间，什么东西可以共享？

可能共享的部分：

  • 视觉表征
  • 语言理解
  • 物体和空间关系
  • 任务阶段
  • 高层skill
  • 失败和恢复模式

不太容易共享的部分：

  • 关节动作
  • 控制频率
  • 夹爪形态
  • 相机标定
  • workspace范围
  • 低层动力学

这就是 robot foundation model 和普通多模态模型的差别。图片和文字比较容易放进同一个token空间，动作却和embodiment绑在一起。

一个 mobile manipulator、一个桌面机械臂、一个双臂机器人，它们看到的世界可能相似，但能做的动作并不一样。

embodiment information

如果把多种机器人数据混在一起，模型需要知道“自己现在是哪种身体”。

可能的条件信息包括：

  • robot type
  • camera setup
  • end-effector type
  • action space definition
  • control frequency
  • workspace limits

否则同一个视觉状态下，模型可能学到互相冲突的动作。

比如同样是“抓杯子”，一个机器人输出末端位姿增量，另一个输出关节速度。如果不告诉模型动作语义，混在一起训练就会变得很奇怪。

所以 embodiment metadata 不是工程细节，它直接决定数据能不能混。

action abstraction

为了跨机器人共享，很多系统会把动作抽象到更高层。

几种常见层次：

  • raw joint command
  • end-effector delta pose
  • waypoint
  • object-centric action
  • skill token
  • language-level subgoal

越底层，执行精度越高，但跨机器人越难共享。越高层，越容易复用，但需要低层控制器把它落地。

这和前面的 hierarchical RL 接得上。foundation model 不一定要直接输出电机控制量，它也可以输出 skill 或 waypoint，让下层系统执行。

video-only data

机器人动作数据很贵，但视频数据相对多。

video-only data 没有动作标签，不能直接做 behavior cloning。但它仍然有价值：

  • 学物体外观
  • 学任务阶段
  • 学人类操作顺序
  • 学场景里的因果线索
  • 学什么变化算任务进展

比如看很多“打开抽屉”的视频，模型也许学不到具体关节命令，但能知道手先靠近把手、拉开、抽屉位置发生变化。

问题是 video-only data 和 robot action 之间有 gap。人手和夹爪不一样，人类身体的 affordance 也不一样。所以 video pretraining 更像提供视觉和任务先验，真正动作还要靠机器人数据对齐。

successful data is not enough

只收成功轨迹会让模型看起来很干净，但真实部署里不够。

成功数据告诉模型怎么做对，但不一定告诉它：

  • 快失败时怎么修正
  • 已经偏离轨迹时怎么回来
  • 哪些动作会造成危险
  • 哪些场景应该停下来请求帮助

失败数据很丑，但很有信息量。尤其是带标注的失败，比如 collision、slip、wrong object、timeout，可以直接指导下一轮数据收集。

这和 evaluation 那篇的想法一致：评估不是最后的报告，而是下一批数据的入口。

data flywheel

一个比较理想的闭环可能是：

  1. 用已有数据训练初始policy
  2. 在受控场景里部署
  3. 记录成功、失败和intervention
  4. 对失败做分类
  5. 补采最缺的场景
  6. 重新训练或微调

这就是 data flywheel。它的关键不只是“多收数据”，而是知道应该收什么数据。

如果模型总是在透明杯子上失败，就补透明杯子；如果总是在遮挡时失败，就补遮挡；如果总是在抓取后滑落，就补恢复和触觉相关数据。

benchmark and reality

robot foundation model 很需要 benchmark，但 benchmark 也很难代表真实部署。

benchmark要可重复，所以场景会被限制；真实部署恰好相反，最麻烦的是那些没被限制住的小变化。

所以我觉得评估最好分两层：

  • controlled benchmark：比较算法，保证可复现
  • deployment evaluation：看真实场景里的失败和恢复

前者让研究能交流，后者让系统真的变好。

只看benchmark分数，很容易把问题简化成“模型又大了一点”。但机器人最后要面对桌面、物体、人、光照、磨损和重置成本，这些都不会因为分数好看就消失。

小结

robot foundation model 这条线可以先记：

  • 难点不只是scale，而是多机器人数据怎么对齐
  • embodiment metadata 很重要，不然动作语义会混
  • action abstraction 决定跨机器人共享的程度
  • video-only data 能提供任务先验，但不能直接替代动作数据
  • 失败数据和intervention 对持续改进很关键
  • data flywheel 比一次性训练更符合真实机器人系统
  • benchmark 要和 deployment evaluation 分开看

如果说 VLA 是把 vision、language、action 接起来，那 robot foundation model 更像是把模型、数据、机器人平台和部署反馈接成一个会慢慢变好的系统。
