Embodied intelligence notes 11 - runtime safety and guardrails

前面写 human-in-the-loop 时说过，安全边界不能只靠人类接管。这里继续往下写 runtime safety。

很多论文会把策略写成：

但真实机器人部署时，中间不会这么裸。policy 输出的动作通常要经过一层或多层 guardrail，最后才会真的发给机器人。

这篇先把这层 runtime system 拆开。

policy is not the whole system

策略模型不是系统边界。

一个机器人应用里，除了 policy，至少还有：

  • perception pipeline
  • state estimator
  • controller
  • safety monitor
  • task planner
  • human interface
  • logger
  • reset / recovery logic

如果只优化 policy success rate，很容易忽略这些部分。但部署时，很多失败不是 policy 本身“不会做”，而是系统没有处理边界情况。

比如相机掉帧、目标物体丢失、夹爪反馈异常、网络延迟，这些都不是普通 benchmark 会认真覆盖的问题。

action constraints

最基础的 guardrail 是动作约束。

比如限制：

  • workspace boundary
  • joint limit
  • velocity
  • acceleration
  • force / torque
  • gripper width
  • end-effector orientation

这类约束不应该交给 policy 自己学。因为一次越界就可能造成真实损失。

但要注意，裁剪动作会改变 policy 的实际执行分布。policy 以为自己输出了一个动作，真实机器人执行的是另一个动作。这个差异需要被记录，否则后续训练数据会混乱。

safety filter

safety filter 比简单 clamp 更进一步。它会根据当前状态判断动作是否允许。

比如：

  • 末端路径是否会碰撞
  • 目标点是否在可达空间
  • 速度是否接近人类
  • 力控状态是否异常
  • 是否即将进入不可恢复姿态

如果动作不安全，可以：

  • reject
  • slow down
  • replace with safe action
  • trigger fallback
  • ask human

这里的关键是：filter 应该在 policy 和 robot 之间有明确权限。policy 可以建议动作，但不能绕过安全层。

watchdog

runtime system 里需要 watchdog。

watchdog 不是智能模块，它更像一组简单但可靠的检查：

  • perception 是否超时
  • controller 是否还在响应
  • robot state 是否更新
  • action command 是否连续异常
  • task 是否卡住太久
  • emergency stop 是否触发

很多真实系统故障不是模型预测错，而是某个子系统停住了。如果没有 watchdog，policy 可能继续输出动作，但世界状态已经不可信。

这类检查不需要复杂，但必须稳定。

fallback

发现异常后不能只说 failed，还需要 fallback。

常见 fallback：

  • stop motion
  • move to safe pose
  • open gripper
  • re-detect scene
  • retry from previous subgoal
  • ask human
  • abort task

fallback 的设计要和任务风险匹配。桌面整理任务可以慢慢停下来；靠近人类的移动机器人则可能需要更严格的急停逻辑。

一个可用系统应该知道什么时候继续，什么时候退回安全状态。

uncertainty gate

前面写 exploration 和 world model 时都提过 uncertainty。部署时，uncertainty 可以变成 gate。

比如：

不是所有不确定都要停机。有些不确定可以通过多看一眼解决，有些可以通过降速解决，有些才需要人类介入。

这比把 uncertainty 当成论文里的额外指标更实际：它应该影响 runtime decision。

logging

安全系统必须记录发生了什么。

至少要记录：

  • raw policy action
  • filtered action
  • filter reason
  • robot state
  • perception confidence
  • intervention event
  • fallback event
  • task outcome

这些日志对后续训练很重要。比如如果很多动作都被 filter 裁掉，说明 policy 学到的动作分布和真实可执行空间不匹配。

没有日志，安全层只是挡住问题；有日志，安全层还能把问题变成下一轮数据。

evaluation of safety layer

safety layer 自己也要评估。

几个问题：

  • 它有没有拦住危险动作？
  • false positive 多不多？
  • 是否让动作变得太保守？
  • 是否引入抖动？
  • 是否影响任务完成率？
  • 是否覆盖真实部署里的主要风险？

过弱的 safety layer 没用；过强的 safety layer 会让机器人什么都不敢做。

所以安全不是越保守越好，而是要把风险压到可接受范围，同时保持任务能力。

relation to learning

runtime safety 不只是工程外壳，也会影响学习。

因为训练数据里看到的是 filtered action 和真实结果。如果不区分 raw action 和 executed action，模型可能学错。

小结

runtime safety 可以先记：

  • policy 不是整个机器人系统
  • action constraints 应该显式存在，不能只靠模型学
  • safety filter 要有权限拦截或替换动作
  • watchdog 负责发现系统级异常
  • fallback 决定异常后怎么回到安全状态
  • uncertainty 可以做 runtime gate
  • 日志要记录 raw action、executed action 和 filter reason
  • safety layer 自己也需要评估

具身智能最后不是把一个模型放到机器人上就结束。真正能跑起来的系统，往往是 policy、controller、safety layer、human feedback 和 logging 一起工作。
