Embodied intelligence notes 14 - benchmark design and reproducibility

前面写 evaluation 时主要看单个策略怎么评估。这篇继续往外走一点：如果要设计一个具身智能 benchmark，应该注意什么？

机器人 benchmark 很难做。NLP 或 vision 里，给一个固定 test set 就能比较很多模型；机器人里，环境、硬件、重置方式、相机位置、物体状态都会影响结果。

所以 benchmark 的价值不只是“排个榜”，更重要的是让结果可解释、可复现。

task definition

第一个问题是任务到底是什么。

比如“把杯子放到盘子上”，至少要定义：

  • 哪些杯子算目标
  • 盘子位置是否固定
  • 杯子初始姿态怎么采样
  • 成功条件是什么
  • 允许推还是必须抓
  • 中途碰到别的物体算不算失败

如果这些不写清楚，两个系统都说 success rate 90%，其实可能不是同一个任务。

任务定义应该尽量具体，但也不能具体到只剩一个固定场景。不然 benchmark 测的是记忆和过拟合，不是泛化。

initial state distribution

初始状态分布很关键。

机器人任务通常不是一个状态，而是一组状态：

这个分布决定任务难度。

比如抓杯子：

  • 杯子永远在桌子中央，很简单
  • 杯子随机放在桌面不同位置，难一点
  • 杯子可能被部分遮挡，更难
  • 杯子和其他物体混在一起，再难一点

如果 benchmark 只给一个固定初始状态，策略很容易学到场景特化技巧。

所以报告结果时最好写清楚 initial state distribution，而不是只写任务名字。

success condition

success condition 也要具体。

比如放置任务：

  • 物体是否必须完全在目标区域内？
  • 是否要保持稳定几秒？
  • 允许轻微倾斜吗？
  • 机器人松手后才算成功，还是抓着也算？
  • 是否允许人类中途介入？

这些细节会显著改变结果。

一个好的 success condition 应该尽量可自动判断，同时符合真实任务意义。如果条件太宽，策略会钻空子；条件太窄，评估又会过于脆。

trial count

机器人实验贵，所以很多论文 trial 数不多。

但 trial 太少时，success rate 很不稳定。比如 10 次里成功 8 次，看起来是 80%，但置信区间其实很宽。

更合理的报告方式至少包括：

  • number of trials
  • success count
  • mean success rate
  • confidence interval 或 bootstrap estimate
  • 每类环境扰动下的单独结果

如果只给一个平均数，很难判断差异是否真实。

failure taxonomy

只报告 success rate 不够，失败类型也要记录。

比如：

  • perception failure
  • planning failure
  • grasp failure
  • collision
  • slip
  • timeout
  • recovery failure
  • unsafe stop

这些失败类型对应不同改进方向。

如果 A 策略失败主要是 perception，B 策略失败主要是 collision，那么它们的下一步工作完全不同。只看总分会丢掉这些信息。

failure taxonomy 也能帮助数据闭环：下一批数据应该补哪类失败，不能靠猜。

intervention protocol

很多真实机器人评估都会有人类介入，但介入协议必须写清楚。

需要定义：

  • 什么时候允许介入
  • 介入后 trial 是否继续
  • 介入算失败还是单独统计
  • soft intervention 和 hard reset 是否区分
  • emergency stop 是否单独计数

如果一个系统成功率高但 intervention 很多，不能和完全自主系统直接比较。

所以 benchmark 里最好把 autonomy 和 success 分开报告：

environment variation

真实部署里的变化很多，benchmark 至少要覆盖一部分。

常见 variation：

  • lighting
  • camera pose
  • object pose
  • distractor objects
  • object category
  • surface material
  • mild occlusion
  • sensor noise

可以分层测试：

  • clean setting
  • mild shift
  • hard shift

这样能看出策略到底是基础能力不足，还是只对分布外变化敏感。

这比混成一个平均分更有信息。

reproducibility log

机器人 benchmark 要可复现，日志很重要。

至少应该记录：

  • task config
  • initial state seed 或采样参数
  • object set
  • camera setup
  • robot model
  • controller config
  • policy checkpoint
  • raw action
  • executed action
  • outcome
  • failure label

这些日志不一定全部公开，但系统内部必须保留。否则一旦结果异常，很难复盘。

尤其是 runtime safety 那篇提到的 raw action / executed action 差异，在 benchmark 里也应该记录。

sim benchmark vs real benchmark

仿真 benchmark 和真实 benchmark 各有价值。

仿真：

  • 便宜
  • 可并行
  • 容易复现
  • 可以测试大量随机状态

真实：

  • 更接近部署
  • 暴露硬件和接触问题
  • 评估 sim-to-real gap
  • 成本高、噪声大

我觉得两者不应该互相替代。仿真适合快速比较方法，真实评估适合确认系统是否真的可用。

如果一个方法只在 sim 里很好，还不能说明能部署；如果只在一个真实场景里很好，也不能说明泛化。

leaderboard risk

benchmark 一旦变成 leaderboard，就会有优化指标的倾向。

这不是坏事，但要小心：

  • 过拟合固定任务
  • 忽略安全和干预成本
  • 只优化平均成功率
  • 少报失败细节
  • 用复杂工程 trick 淹没方法本身

所以 benchmark 最好不要只有一个总分。可以分成几个维度：

  • task success
  • robustness
  • autonomy
  • safety
  • recovery
  • efficiency

这样结果更难被一个数字掩盖。

小结

具身智能 benchmark 可以先记：

  • 任务定义、初始状态分布和成功条件必须写清楚
  • trial count 太少时，success rate 信息有限
  • failure taxonomy 比单一平均分更有用
  • intervention protocol 要独立报告
  • environment variation 应该分层测试
  • reproducibility log 是后续复盘的基础
  • sim benchmark 和 real benchmark 互补
  • leaderboard 不应该只看一个总分

机器人评估最怕“数字看起来很干净，但不知道发生了什么”。好的 benchmark 应该让失败也有信息，让下一步改进有方向。
